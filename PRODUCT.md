# Product

## Register

brand

## Users

Primary visitors are recruiters and hiring managers evaluating Himanshu Shekhar's credibility quickly from a GitHub profile README. Secondary visitors include technical collaborators and open-source peers looking for proof of taste, project direction, and approachability. They arrive in a fast-scanning context and need a clear read on public project work: `CardsAgainstBhayanak`, `BhayanakCast`, `BhayanakBot`, `AntSimulation`, and `hshekhar.bhayanak.net`.

## Product Purpose

This repository powers a public GitHub profile/portfolio README. Its job is to extend the personal site's terminal-native identity into GitHub while keeping the README readable as Markdown. Success means a visitor can understand the public project focus, technical range, portfolio link, and contact path within a short scan, while still getting the same playful CLI personality as `hshekhar.bhayanak.net`.

## Brand Personality

Playful, technical, sharp. The voice should feel like a competent engineer's terminal session with a mischievous streak: memorable and hackerish, but still readable, credible, and intentional. Humor should sharpen the message, not replace substance.

## Anti-references

Avoid generic badge-wall profiles where shields and stats carry the whole identity. Avoid corporate resume-page tone: dry, buzzword-heavy, and interchangeable. Avoid unreadable terminal cosplay: tiny text, low contrast, hidden links, motion gimmicks, or jokes that block scanning. Avoid overanimated or noisy portfolio treatment that distracts from credibility, accessibility, or the GitHub-native reading experience.

## Design Principles

1. Preserve the terminal frame: reuse prompt/output rhythm from `hshekhar.bhayanak.net` where Markdown can support it cleanly.
2. Lead with public proof, not decoration: every visual or content choice should help a visitor understand the focused public projects faster.
3. Keep GitHub-native readability: the README should work as Markdown first, with terminal flavor as structure rather than fragile presentation tricks.
4. Make personality useful: jokes and playful phrasing should clarify the engineering point of view, not become filler.
5. Preserve a clear contact path: visitors should never wonder how to follow up.

## Accessibility & Inclusion

Target WCAG AA for future design work. Maintain readable contrast for text and badges where controllable, avoid relying on color alone, keep links descriptive, preserve logical heading order, and respect reduced-motion expectations for any generated or embedded motion where the implementation allows control.
